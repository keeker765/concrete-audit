# 混凝土试块造假识别系统 — 技术框架

> **核心问题**：施工现场拍湿试块（刚制作），送检时换成干的高质量假试块。  
> **目标**：通过图像匹配，判断「注册时的湿试块（现场照）」与「送检时的干试块（压测照）」是否为同一块。

## 零、真实数据概况（象山试块照片.zip）

| 项目 | 数值 |
|------|------|
| 企业数 | 6 家 |
| 批次总数 | 90 批 |
| 现场照（.jpeg，湿） | 270 张 |
| 压测照（.jpg，干） | 270 张 |
| 每批结构 | 3 块试块，各 1 张现场照 + 1 张压测照 |

- `.jpg`（纯时间戳名）= **压测照**，较晚时间戳，侧视，试块在压机中，干燥

**目录结构：**
```
data/
  {企业名}/
    砼-{工地}-压-{编号}/
      image_*.jpeg   ← 现场照（3张，wet，较早）
      *.jpg          ← 压测照（3张，dry，较晚）
```

## 一、问题本质分析（基于实际图片）

### 系统现状：甬砼码已有 QR 追踪

现有「甬砼码」蓝色圆形贴纸（直径约 5cm）贴在每块试块的同一面：
- 编号如 `011250004384-1/2/3`，一批3块各有尾号
- 现场照和压测照中均可见**同一枚贴纸**
- 图片嵌入水印：拍摄时间、企业名、GPS坐标、强度等级

**现有系统的漏洞（造假方式）：**
```
① 揭下原贴纸，贴到预备好的高强度假试块上
② 打印假贴纸（号码可从网络获取）
```

### 图像识别的价值：贴纸 ≠ 混凝土本体

QR 贴纸验证身份，图像匹配验证**贴纸下面那块混凝土**是否吻合。

### 实际图片的关键观察

| | 现场照（.jpeg，湿） | 压测照（.jpg，干） |
|--|--|--|
| 视角 | **俯视**（模具顶面向上） | **侧视**（压机中侧面朝前） |
| 颜色 | 深灰、反光、湿润 | 浅灰、哑光、部分碎裂 |
| 贴纸位置 | 顶面（模具开口面） | 侧面（**同一物理面**，试块旋转了） |
| 特征源 | 气孔、骨料凸起、表面纹路 | 气孔（更清晰）、骨料斑点 |

> ⚠️ 压测照是**压机中**拍摄，试块边角已碎裂，但贴纸所在面基本完整。

### 为什么可做？

**气孔 + 骨料空间分布是不变量** ——  
每块试块的气孔随机分布，干湿状态、28天养护均不改变其空间位置。

**天然标定物：圆形贴纸**  
贴纸为已知正圆（约5cm直径），倾斜拍摄时呈椭圆：
```
椭圆 → 已知为正圆 → 反算 homography → 透视矫正 → 标准化视角
```
这让干湿两张不同视角的图可以对齐到同一坐标系，是最大技术亮点。

## 三、各阶段技术方案

### Stage 1：图像采集规范（现有系统已满足）

- 现场照：甬砼码 App 拍摄，含 GPS + 时间戳水印，俯视进模具顶面
- 压测照：实验室拍摄，侧视试块贴纸面
- **关键要求**：贴纸面清晰可见，无遮挡

### Stage 2：贴纸检测 + 透视矫正（新增）

```python
import cv2
import numpy as np

def detect_sticker_and_rectify(img):
    """检测蓝色圆形贴纸，利用圆形做透视矫正"""
    # Step 1: HSV颜色空间提取蓝色区域
    hsv = cv2.cvtColor(img, cv2.COLOR_BGR2HSV)
    mask = cv2.inRange(hsv, (95, 80, 80), (130, 255, 255))  # 蓝色范围
    
    # Step 2: 检测椭圆轮廓（贴纸在倾斜视角下呈椭圆）
    contours, _ = cv2.findContours(mask, cv2.RETR_EXTERNAL, cv2.CHAIN_APPROX_SIMPLE)
    sticker = max(contours, key=cv2.contourArea)
    ellipse = cv2.fitEllipse(sticker)  # (center, (MA, ma), angle)
    
    # Step 3: 已知贴纸是正圆 → 椭圆→圆的 homography → 透视矫正
    cx, cy = ellipse[0]
    a, b = ellipse[1][0]/2, ellipse[1][1]/2  # 长短半轴
    # 计算仿射变换，将椭圆区域矫正为正圆（即等比缩放短轴）
    # 然后以贴纸为中心裁取 3x 半径的混凝土区域
    
    return rectified_region, sticker_center
```

### Stage 3：预处理（消除干湿差异）

```python
def preprocess(img_region):
    gray = cv2.cvtColor(img_region, cv2.COLOR_BGR2GRAY)
    # CLAHE: 增强局部骨料/气孔对比度，消除全局亮度差
    clahe = cv2.createCLAHE(clipLimit=3.0, tileGridSize=(8, 8))
    return clahe.apply(gray)
```

### Stage 4：特征匹配（Core Module）

#### 方案 A：LightGlue + SuperPoint（**推荐首选**）

- **GitHub**：`cvg/LightGlue`（⭐ 4422）
- **优势**：比 SuperGlue 快 4-10x，对低纹理/外观变化场景有自适应剪枝

# 输入：透视矫正后贴纸周围区域（同一物理面，已去干湿差异）

- `pretrained='indoor'` 权重对低对比度室内场景更鲁棒

#### 已有相关论文验证

> *"Research on Low Contrast Feature Extraction and Registration Effect of Concrete Structure based on SuperGlue Algorithm"*

### Stage 5：判定逻辑

```python
def is_same_specimen(matches, match_conf, threshold=0.65):
    if len(matches) < 15:           # 匹配点太少 → 拒绝
        return False, 0.0
    
    score = match_conf.mean().item()              # 平均置信度
    # RANSAC 检验空间一致性，获得 inlier_ratio
    final_score = 0.6 * score + 0.4 * inlier_ratio
    return final_score > threshold, final_score
```

| 指标 | 含义 | 建议阈值 |
|------|------|---------|
| `match_count` | 有效匹配点数 | ≥ 15 对 |
| `mean_conf` | 平均匹配置信度 | ≥ 0.6 |
| `inlier_ratio` | RANSAC 内点率 | ≥ 0.5 |
| `final_score` | 综合评分 | ≥ 0.65 |

## 七、可解释性设计（Explainability）

> 造假识别不是黑箱分类——每个判定结果必须让**非技术背景的监管人员**看得懂、信得过、能举证。

### 7.1 为什么可解释性在本项目中是硬需求

| 场景 | 要求 |
|------|------|
| 行政执法 | 住建局出具处罚决定，需要附带可视化证据 |
| 法律举证 | 如进入司法程序，「AI 说不一样」不能作为证据，需要**可理解的比对图** |
| 监管人员信任 | 检测站工作人员需要看懂匹配结果，才会使用系统 |
| 申诉回溯 | 被误判的企业有权看到「系统为什么判定为调包」的完整依据 |
| 持续改进 | 开发者需要理解误判原因，调整算法参数 |

### 7.2 可解释性的三个层级

#### 层级一：视觉匹配图（Visual Match Overlay）——最直观

在两张图上画出匹配点对的连线，用颜色表示置信度：

```python
import matplotlib.pyplot as plt
import numpy as np

def draw_match_explanation(img_wet, img_dry, kpts0, kpts1, matches, conf):
    """
    生成可解释的匹配可视化图
    绿线 = 高置信匹配点，黄线 = 中等，红线 = 低置信 / RANSAC 外点
    """
    fig, axes = plt.subplots(1, 2, figsize=(20, 10))
    axes[0].imshow(img_wet); axes[0].set_title('现场照（湿）')
    axes[1].imshow(img_dry); axes[1].set_title('压测照（干）')
    
    for i, (m, c) in enumerate(zip(matches, conf)):
        color = 'green' if c > 0.8 else 'orange' if c > 0.5 else 'red'
        # 在两图对应位置画圆 + 连线
        p0, p1 = kpts0[m[0]], kpts1[m[1]]
        for ax, pt in [(axes[0], p0), (axes[1], p1)]:
            ax.plot(pt[0], pt[1], 'o', color=color, markersize=6)
    
    plt.suptitle(f'匹配点: {len(matches)} 对 | 平均置信度: {conf.mean():.2f}')
    return fig
```

**输出效果**：一张并排图，左现场照右压测照，绿色点表示「这些气孔/骨料位置吻合」

> 监管人员用法：「看，左图的这个气孔和右图的这个气孔对上了，20 多个点都对上了，说明是同一块。」

#### 层级二：数值分解报告（Score Decomposition）——可量化

将 `final_score` 拆解成人类可理解的子维度：

```python
def generate_score_report(matches, match_conf, inliers_mask):
    report = {
        '匹配点数': len(matches),
        '高置信匹配(>0.8)': int((match_conf > 0.8).sum()),
        '平均置信度': round(match_conf.mean().item(), 3),
        'RANSAC_内点数': int(inliers_mask.sum()),
        'RANSAC_内点率': round(inliers_mask.sum() / len(matches), 3),
        '综合评分': round(final_score, 3),
        '判定结果': '✅ 同一块' if final_score > 0.60 else '❌ 疑似调包',
    }
    
    # 自然语言解释
    if report['判定结果'].startswith('✅'):
        report['解释'] = (
            f"共发现 {report['匹配点数']} 个特征匹配点，"
            f"其中 {report['高置信匹配(>0.8)']} 个为高置信度匹配，"
            f"空间一致性检验通过率 {report['RANSAC_内点率']*100:.0f}%，"
            f"综合评分 {report['综合评分']}（阈值 0.65），判定为同一试块。"
        )
    else:
        report['解释'] = (
            f"仅发现 {report['匹配点数']} 个匹配点"
            f"（高置信仅 {report['高置信匹配(>0.8)']} 个），"
            f"空间一致性仅 {report['RANSAC_内点率']*100:.0f}%，"
            f"综合评分 {report['综合评分']} 低于阈值 0.65，"
            f"**疑似试块被调包**，建议人工复核。"
        )
    return report
```

#### 层级三：特征热力图（Feature Attention Heatmap）——可定位

标注「系统在看哪里」，让人知道判定依据是混凝土表面而非贴纸或水印：

```python
def generate_attention_heatmap(img, keypoints, conf):
    """
    将关键点置信度渲染为热力图叠加在原图上
    红色高亮区域 = 系统判定依据的主要特征区域
    """
    heatmap = np.zeros(img.shape[:2], dtype=np.float32)
    for kp, c in zip(keypoints, conf):
        cv2.circle(heatmap, (int(kp[0]), int(kp[1])), 
                   radius=15, color=float(c), thickness=-1)
    heatmap = cv2.GaussianBlur(heatmap, (51, 51), 0)
    heatmap = cv2.normalize(heatmap, None, 0, 255, cv2.NORM_MINMAX)
    colored = cv2.applyColorMap(heatmap.astype(np.uint8), cv2.COLORMAP_JET)
    return cv2.addWeighted(img, 0.6, colored, 0.4, 0)
```

**用途**：
- 红色集中在气孔/骨料 → 系统基于物理特征判断 → 可信
- 红色集中在贴纸区域 → 系统在匹配贴纸而非混凝土 → 需 mask 贴纸
- 红色集中在水印文字 → 预处理有问题 → 需去水印

### 7.4 可解释性 vs 准确性的关系

本方案在可解释性上有**天然优势**，因为使用的是特征点匹配而非端到端黑箱分类：

| 方法类型 | 可解释性 | 准确性 | 本项目选择 |
|---------|---------|--------|-----------|
| 端到端 CNN 分类 | ❌ 黑箱，只输出概率 | 高（需大量数据） | 不选 |
| Siamese Network | ⚠️ 需 Grad-CAM 辅助 | 高（需配对数据） | 长期可选 |
| **特征点匹配 (LightGlue)** | **✅ 天然可解释** | 中高 | **首选** |
| 人工比对 | ✅ 完全可解释 | 低（人眼难以跨干湿比对） | 辅助 |

> **关键洞察**：LightGlue 每个匹配点都有明确的空间坐标和置信度，  
> 这意味着每个判定结论都可以回溯到具体的像素位置——  
> 「因为**这个气孔**和**那个气孔**对上了」，而不是「因为模型说了 0.82 分」。

### 7.5 异常判定时的人机协作流程

```
系统判定为 ❌ 疑似调包
        ↓
自动生成报告（含匹配可视化 + 分数分解 + 热力图）
        ↓
推送给监管人员（App / 邮件 / 系统内通知）
        ↓
人工复核（看图 + 看匹配点 + 看分数）
    ├── 确认：启动调查流程
    ├── 误判：标记为误判，反馈给系统（扩充训练集）
    └── 不确定：要求重新拍摄 / 补拍其他面
```

**绝对不做**：系统不直接出处罚决定。所有「疑似调包」判定只作为**预警**，最终由人工复核确认。

## 九、实验报告 — V2 Pipeline 完整流程与结果

> 本章详细说明 demo_v2.py 的完整处理流水线，以及在 18 对样本上的实测结果。

### 9.1 一对图像的完整处理流程

每一对干湿图像经过以下 8 个步骤。以「宁波象保#1」为例：

```
输入: 
  湿态照 (wet): 宁波象保_251978_wet1.jpeg  1920×2560px (portrait)
  干态照 (dry): 宁波象保_251978_dry1.jpg   2560×1440px (landscape)
```

#### Step 1: 加载与缩放
```
目的: 控制计算量，避免过大图像
操作: max(H,W) > 1024px 时等比缩放
结果: wet → 768×1024   dry → 1024×576
```

#### Step 2: 贴纸检测（HSV 蓝色圆形）
```
目的: 找到甬砼码贴纸的位置和大小
操作: 
  1. 转 HSV 色彩空间
  2. 蓝色掩膜: H=[90,135] S=[50,255] V=[50,255]
  3. 形态学开闭运算去噪
  4. 找最大蓝色轮廓 → 拟合椭圆 → 得到中心点 + 长短轴 + 角度
结果: 
  wet: center=(530,430), radius≈256px, aspect_ratio=0.98
  dry: center=(512,330), radius≈128px, aspect_ratio=0.99
```

#### Step 3: ROI 裁切 [核心改进]
```
目的: 只保留贴纸周围的混凝土面，消除背景干扰
操作: 以贴纸中心为原点，裁切 2.5×贴纸半径 的正方形区域
效果:
  wet: 768×1024 → 614×666 (只剩混凝土面 + 贴纸)
  dry: 1024×576 → 306×306 (只剩混凝土面 + 贴纸)
消除了: 压力机金属框架、模具壁、桌面、水印文字
```

> **为什么 2.5 倍？** 贴纸直径约 5cm，混凝土试块面 15cm。  
> 2.5 × 半径 ≈ 6.25cm 每侧 → 总覆盖 12.5cm，基本覆盖试块面。

#### Step 4: 透视矫正（椭圆→圆）[V2.5 迭代 Homography]
```
目的: 消除倾斜拍摄造成的透视变形
核心改进: 使用 Homography (8 DOF) 替代仿射 (6 DOF)

原理 (来自 SO "How to rectify a detected ellipse"):
  - 透视投影下圆的轴心 ≠ 检测到的椭圆的轴心
  - 仿射变换只能做线性拉伸，无法完全还原透视形变
  - Homography 用椭圆4端点→圆4端点计算透视矩阵

操作:
  1. 检测贴纸椭圆 → fitEllipse → 长短轴比 (ratio)
  2. ratio > 0.99 → 跳过
  3. ratio ≤ 0.99 → 取椭圆4端点（长轴两端 + 短轴两端）
     → 映射到同心圆的4端点（半径=长半轴）
     → getPerspectiveTransform 计算 Homography 矩阵
     → warpPerspective 应用
  4. 迭代: 重新检测 → 若 ratio 仍 < 0.99 → 再次矫正（最多 3 轮）

效果对比:
  象山甬港#3: 0.820 → affine_iter=0.985 → homography_iter=0.997
  象山县美#3: 0.927 → affine_iter=0.998 → homography_iter=0.995
  宁波大目#1: 0.990 → homography 使匹配从 19→209 点 (score 0.326→0.874)
```

#### Step 5: CLAHE 预处理
```
目的: 消除干湿状态的亮度差异
操作:
  1. 转灰度
  2. CLAHE 自适应直方图均衡化 (clipLimit=3.0, 8×8网格)
  3. 灰度→3通道（LightGlue 需要 RGB 输入）
效果: 湿态(深灰反光) 和 干态(浅灰哑光) 对比度拉平
```

#### Step 6: 贴纸 Mask 生成
```
目的: 在匹配时排除贴纸区域——我们要匹配的是混凝土纹理，不是贴纸图案
操作: 在裁切后的 ROI 上重新检测贴纸 → 生成二值 mask
注意: ROI 裁切后水印已被自然排除，不需要额外的水印 mask
```

#### Step 7: 旋转对齐 [V2.3 混合策略]

```
目的: 消除试块在模具→压机之间的任意旋转

[V2.3 混合策略] 暴力保底 + QR 精确增强:
  始终包含:
    暴力 0°/90°/180°/270° (rot90, 无插值损失) → 4 个候选
  
  如果两张图都检测到 QR 码角度:
    计算 QR 角度差 → 2 个精确候选（diff 和 diff+180°）
    仅当精确角度偏离最近 90° ≥ 3° 时才加入候选（否则暴力已覆盖）
    → 总共 4~6 个候选
  
  取所有候选中得分最高的

为什么用混合策略？（V2.2 教训）:
  - 纯 QR 方案在透视矫正后 QR 检测不稳定（如日升#2 检测到 287° 而正确是 180°）
  - 暴力搜索保底，永远不退化
  - QR 精确角度能处理非 90° 旋转（如拍照角度差异导致的 175.4° 或 6°）

QR 检测增强（8 种预处理，检测率 100%）:
  原图 → CLAHE → 锐化 → OTSU二值化 → 放大2× → 放大2×+CLAHE → 自适应阈值 → 放大2×+锐化
  经测试: 原始阶段86% → 缩放阶段81% → ROI阶段94% → 矫正阶段100% (36/36)

透视矫正阈值: 椭圆长短轴比 < 0.99 即矫正（更激进，确保 QR 角度准确）

可视化一致性:
  匹配时保存旋转后的实际图像 (img0_vis)，可视化直接使用，不重新旋转

代价: ~20s/对 (CPU) / ~0.7s/对 (GPU) — 4~6 candidates
```

#### Step 8: 评分与判定

**术语解释：**

| 英文缩写/术语 | 全称 | 中文含义 |
|---------------|------|----------|
| **RANSAC** | Random Sample Consensus | 随机抽样一致性算法 — 从匹配点对中剔除错误匹配（外点），只保留几何关系一致的正确匹配（内点）。通过反复随机抽样来拟合一个变换模型（Homography），最终统计有多少点符合这个模型 |
| **Homography** | Homography Matrix | 单应性矩阵 — 一个 3×3 矩阵，描述两张图像中同一个平面的透视变换关系。如果两张照片拍的是同一个试块面，它们之间就存在一个 Homography |
| **Inlier** | Inlier point | 内点 — 符合 Homography 变换的匹配点，即被 RANSAC 判定为"正确匹配"的点 |
| **Inlier Ratio** | Inlier Ratio | 内点率 — 内点数 ÷ 总匹配数。同一块试块的干湿照理论上内点率接近 1.0；不同试块则很低 |
| **Match Confidence** | Match Confidence Score | 匹配置信度 — LightGlue 神经网络对每对匹配点给出的"我有多确信这两个点是同一个位置"的分数，范围 [0, 1] |
| **SuperPoint** | SuperPoint (论文名) | 一种深度学习关键点检测器，能在图像中找到角点、纹理交叉点等有辨识度的位置，并为每个点生成 256 维描述向量 |
| **LightGlue** | LightGlue (ICCV 2023) | 一种轻量级神经网络匹配器，输入两组 SuperPoint 特征，输出匹配点对及置信度。比传统最近邻匹配更准确 |
| **CLAHE** | Contrast Limited Adaptive Histogram Equalization | 限制对比度自适应直方图均衡化 — 把干（浅灰）和湿（深灰）的图像亮度拉到相近水平，让后续匹配不受颜色干扰 |
| **ROI** | Region of Interest | 感兴趣区域 — 这里指以贴纸为中心裁切出的混凝土表面区域，排除背景（压力机、桌面等） |
| **MIN_MATCHES** | Minimum Matches | 最低匹配数阈值 (=15) — 如果过滤后的匹配点少于此值，认为证据不足，不做判定 |
| **QR 定向** | QR Code Orientation | 利用 QR 码的角点位置计算贴纸在图中的朝向角度，用于精确旋转对齐两张图 |

**评分公式：**

```
raw_score = 0.5 × 平均匹配置信度(mean_conf) + 0.5 × RANSAC内点率(inlier_ratio)

[V2.1] 匹配数惩罚:
  若 filtered_matches < MIN_MATCHES (15):
    final_score = raw_score × (matches / MIN_MATCHES)
    例: 4 个匹配 → penalty = 4/15 = 0.27 → 分数大幅降低
  否则:
    final_score = raw_score

判定 (三分类):
  filtered_matches < MIN_MATCHES → INSUFFICIENT（证据不足，需人工复核）
  final_score > 0.60 → SAME（判定为同一块试块）
  final_score ≤ 0.60 → DIFFERENT（疑似调包）
```

**为什么用这两个指标？**
- **匹配置信度**衡量"特征层面"两张图有多像 — 高说明找到的对应点质量好
- **RANSAC 内点率**衡量"几何层面"这些匹配点是否空间一致 — 高说明匹配点不是随机碰巧

**示例（宁波象保#1）：**
```
最佳旋转: -3.1° (QR 定向)
过滤后匹配点: 297 对 (≥ 15, 无惩罚)
RANSAC 内点: 295 (99.3%)
平均置信度: 0.628
综合评分: 0.5×0.628 + 0.5×0.993 = 0.810 → SAME ✅
```

**反例（宁波大目#1）：**
```
过滤后匹配点: 13 对 (< 15, 惩罚 = 13/15 = 0.867)
raw_score = 0.5×0.318 + 0.5×0.615 = 0.467
final_score = 0.467 × 0.867 = 0.405 → INSUFFICIENT ⚠️
```

## 十、现有方案调研

### 10.1 直接相关专利

**CN113158928A — 一种基于图像识别的混凝土试块防造假方法**
- 发明人: 陈维苹（浙江云仪科技有限公司）
- 申请日: 2021-04-27，授权日: 2023-09-19，状态: **有效**
- **核心思路**：与我们完全一致 — 现场拍照 P1 + 送检拍照 P2 → 图像比对

**该专利的技术方案：**
1. 将 P1/P2 缩放到 **256×256**
2. 对灰度图做**二值化边界提取** → 取边界点最多的 4 个区块作为模板
3. 用 **opencv 模板匹配** (template matching) 做相似度计算
4. 旋转搜索范围: **-15° 到 +15°**（每度一次）
5. 自动裁剪: **霍夫变换**提取试块边界直线 → 裁出正方形
6. 声称平均准确率 **95%**

**与我们方案的对比：**

| 维度 | CN113158928A (专利) | 本方案 (SuperPoint+LightGlue) |
|------|---------------------|-------------------------------|
| 特征提取 | 二值化边界 + 模板匹配 | 深度学习关键点 + 神经匹配器 |
| 分辨率 | 256×256 | 1024-1536 |
| 旋转处理 | ±15° 连续搜索 | 0°/90°/180°/270° 离散搜索 |
| ROI 定位 | 霍夫变换找直线边界 | 蓝色贴纸检测定位 |
| 干湿鲁棒性 | 未特别处理 | CLAHE 自适应均衡化 |
| 透视矫正 | 无 | 椭圆→圆仿射变换 |
| 匹配方式 | 滑窗模板匹配 | 局部特征点匹配 + RANSAC |
| 优势 | 简单快速 | 对变形/光照/旋转更鲁棒 |
| 劣势 | 对干湿差异敏感 | 计算量大(~15s/pair on CPU) |

> **关键发现**：该专利使用传统模板匹配，对干湿变化的鲁棒性存疑。
> 我们采用的深度学习特征（SuperPoint）天然对亮度变化更具鲁棒性，
> 结合 CLAHE 预处理，理论上在干湿跨域匹配上有明显优势。

## 十一、当前技术路线总结

### 11.1 技术栈

```
┌─────────────────────────────────────────────────────┐
│                    输入层                             │
│  湿态照 (.jpeg, 1920×2560)  干态照 (.jpg, 2560×1440) │
└──────────────────────┬──────────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────────┐
│                  预处理层                             │
│  1. 缩放 (wet→1024, dry→1536)                       │
│  2. HSV 蓝色贴纸检测 → 中心定位                       │
│  3. ROI 裁切 (2.5×贴纸半径)                          │
│  4. 透视矫正 (椭圆→圆, 需要时)                       │
│  5. CLAHE 亮度归一化                                 │
│  6. 贴纸 Mask 生成                                   │
└──────────────────────┬──────────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────────┐
│                  匹配层                              │
│  × 4 旋转方向 (0°/90°/180°/270°):                   │
│    SuperPoint → 2048 关键点 + 256D 描述子            │
│    LightGlue → 稀疏匹配 + 置信度                     │
│    过滤贴纸区域匹配点                                 │
│    RANSAC Homography → 内点率                        │
│  选最高分旋转                                        │
└──────────────────────┬──────────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────────┐
│                  决策层                              │
│  raw_score = 0.5×置信度 + 0.5×内点率                 │
│  if matches < 15: score × (matches/15) → 惩罚       │
│  INSUFFICIENT / SAME / DIFFERENT                     │
└─────────────────────────────────────────────────────┘
```

### 11.2 当前性能

| 指标 | 数值 |
|------|------|
| 特征提取器 | SuperPoint (max 2048 keypoints) |
| 匹配器 | LightGlue (ICCV 2023) |
| 正确匹配率 | 12/18 (67%) — 需要提高 |
| INSUFFICIENT | 1/18 (6%) — 匹配点不足 |
| 跨批次误报 | 0/6 (0%) — 安全 |
| 阈值 | 0.60 |
| 最低有效匹配数 | 15 |
| 单对耗时 | **~0.69s (GPU)** / ~20s (CPU) |
| GPU | NVIDIA RTX 5070 Laptop GPU (8GB VRAM, SM 12.0) |
| 使用环境 | Python 3.13 + PyTorch 2.11.0+cu128 |
| QR 检测率 | 36/36 (100%) — 矫正阶段，8种预处理 |

### 11.3 已知局限

1. **低纹理表面**：高标号混凝土表面光滑，特征点少 → 宁波大目 3/3 失败
2. **小 ROI 干态照**：部分干态照中试块占比小 → 象山港浦 ROI 仅 364px，3/3 失败
3. **CLAHE 参数固定**：clipLimit=3.0 对所有图像一样，未自适应
4. **QR 180° 模糊性**：QR 码本身有 180° 旋转对称性，需尝试 diff 和 diff+180° 两个候选
